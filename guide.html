How to use Lernexa

From "press play" to "add your own algorithm" — three levels, pick where you are.

Use it Read it Build it

1 Use it

Open any algorithm from the Library, then drive it with the control bar. Nothing to install — it runs in the browser.

Play / Pause

Start the animation, or pause it mid-run to inspect the current state. Shortcut: Space.

Step back / forward

While paused, rewind and replay frame by frame — the picture and the counters/variables jump to that exact step. ← →.

Disabled for a few views (trees, maze, the math demos) where rewinding isn't supported — they stay greyed out rather than pretending to work.

Speed

Slide from 0.25× to 2×. It changes the delay between steps, so fast runs stay smooth and slow runs are easy to follow.

Custom input

Type your own array, target, expression, or numbers, then Apply. Sorting views also generate edge cases (reversed, nearly-sorted, few-unique…). Reset with R.

2 Read it

Every view speaks the same visual language. Once you know the colours and the dashboard, you can read any of the 36 algorithms.

Colour legend

Comparingthe elements / cell being looked at right now
Visitedalready processed (graph nodes, swept cells)
Path / activethe chosen path, pivot, or current pointer
Done / foundsorted, target found, or goal reached
Goal / errorthe end node, or an invalid operation

Counters

The tiles tally real work — comparisons, swaps, array writes, nodes visited. Watch them grow to feel a complexity class.

Variable watcher

The live values of the algorithm's key variables (pointers, indices, accumulators) update as it runs — and rewind with the step buttons.

Step log

A running narration of each decision: why a pointer moved, which branch was taken, what a cell resolved to.

Complexity strip

Best / average / worst time and space for the algorithm — the theory next to the behaviour you're watching.

3 Build it · for the curious

Lernexa is data-driven. data/algorithms.json is the source of truth — it declares each algorithm's metadata, default input, complexity, and the counters/variables its dashboard shows. Three layers stay cleanly separated:

The architecture

Router reads ?algo=<id>, validates it, and boots the workspace. Workspace is the orchestrator — it wires the controls to the engine and the dashboard. Engines (nine of them) do rendering only — DOM, SVG, or Canvas — with zero algorithm logic. Runners are pure algorithm functions that never touch the DOM.

The callback contract

A runner is one async function. It drives the UI only through callbacks — it never reaches into the page:

async function runX(opts) {
  var engine   = opts.engine;       // rendering engine for this algorithm
  var control  = opts.control;      // { isPaused, isAborted } — cooperative
  var getDelay = opts.getDelay;     // await sleep(getDelay()) between frames
  var onLog    = opts.onLog;        // (type, message) → step log
  var onCnt    = opts.onCounter;    // (key [, by])     → bump a counter
  var onVar    = opts.onVarUpdate;  // (name, value)    → variable watcher

  // opts also carries parsed input: array, target, k, customInput, opts.algo…

  // …do work, calling onCnt/onVar/onLog and awaiting sleep(getDelay())…
  return true;   // true = finished, false = aborted
}

The counter/variable keys a runner emits must match the counters/variables it declares in the JSON — a contract the headless tests lock down.

Add an algorithm — mostly data + one function

  1. Declare it — one entry in data/algorithms.json (id, name, category, engine, input, complexity, variables, counters).
  2. Write the runner — js/algorithms/<category>/<name>.js exporting async function runX(opts) (the contract above).
  3. Register it — add it to the RUNNERS map in workspace.js and a <script> tag in workspace.html.

Reuse an existing engine and you write no rendering code at all.
